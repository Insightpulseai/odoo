# Expense Liquidation Agent - Plan

## 1. System Architecture
The data flow follows a strictly channelized lane:

`Telegram -> n8n Trigger/Webhook -> Normalization & Enrichment -> Odoo Liquidation API/Model -> Foundry Extraction & Audit -> Policy Engine -> Approver / Finance Actions`

## 2. Canonical Contracts: Submission Envelope
To decouple Odoo from specific channel quirks, we define a **Submission Envelope**:
- **Envelope ID**: UUID generated by n8n or Odoo.
- **Channel**: `telegram` | `web` | `email`.
- **Sender Identity**: Original UID (e.g., Telegram Chat ID).
- **Employee Mapping**: Status of link between sender and Odoo `hr.employee`.
- **Attachments**: Binary data + original metadata.
- **Extracted Text/Artifacts**: Raw OCR data before Odoo normalization.
- **Expense Lines**: Proposed line data (Date, Amount, Category).
- **Replay Metadata**: Original raw payload for traceability.

## 3. Integration Boundaries
- **Telegram**: Purely an ingress channel. No business logic resides in the Telegram bot.
- **n8n**: Acts as a stateful orchestrator and protocol adapter. It ensures the Odoo API receives a clean `Submission Envelope`.
- **Odoo**: The System of Record for domain truth. Holds the state machine, policy rules, and accounting logic.
- **Foundry (ipai_foundry)**: An external intelligence service consumed by Odoo for extraction and policy-assist flags.

## 4. Failure Handling & Retry Behavior
- **Duplicate Message Delivery**: n8n/Odoo use `source_message_id` for deduplication.
- **Low Confidence Extraction**: Foundry results with confidence < 0.8 automatically trigger `document_state = review_needed`.
- **Mapping Failure**: If sender ID cannot be mapped to an employee, n8n sends an "Enrollment Link" back to the user instead of dropping the submission.
- **Webhook Environment Risk**: Because Telegram supports only one webhook per bot, we mandate:
    - Dedicated **Production Bot** and **Staging Bot**.
    - Isolated n8n instances (or separate paths) to avoid prod-stage collision.

## 5. Security & Compliance
- **Webhook Verification**: n8n validates Telegram secrets before processing.
- **Least-Privilege API**: Odoo service account for n8n is restricted to `hr.expense.liquidation` and relevant attachments.
- **PII Sensitivity**: OCR logs in Foundry are transient; Odoo remains the vault for PII (Employee IDs, TINs).
- **Audit Evidence**: The `Submission Envelope` is attached to the final Odoo record as immutable proof of ingestion.

## 6. Observability
- **Correlation ID**: Passed from n8n to Odoo to Foundry.
- **Stage Metrics**: Tracking time-to-extract and time-to-audit for performance monitoring.
