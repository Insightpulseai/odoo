## Key Observations

1. **Three of four user-facing auth flows are non-functional** (Gmail add-on Microsoft, Google, and API key). Only Odoo local password login works.
2. **All OAuth flows depend on two unbuilt modules**: `ipai_auth_oidc` (Entra provider) and `ipai_mail_plugin` (bridge controller).
3. **Infrastructure auth is well-designed**: Key Vault, managed identity, env-var injection, proxy mode -- all correctly configured.
4. **The auth state nonce system in the Gmail add-on is well-implemented** but untestable without backend endpoints.
5. **Host identity (Gmail) vs ERP identity separation is correctly modeled** in the add-on code and documentation.
6. **Developer environment surfaces are multi-identity**: VS Code binds Azure ML (`proj-ipai-claude`), GitHub, Azure CLI, Docker, and AzDO credentials concurrently. Each extension delegates to its own identity authority (Entra, GitHub, local OS). No cross-extension credential leakage observed, but extension-sourced tokens persist in OS keychain.
7. **Docker Desktop context (`docker-desktop`, `default` profile)** is the active local runtime. ACR auth (`acripaiodoo`) is separate from Docker socket auth and requires explicit `az acr login` or `docker login`.
